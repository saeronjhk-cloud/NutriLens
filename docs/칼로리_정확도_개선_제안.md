# NutriLens 칼로리 정확도 개선 제안서

**작성일**: 2026-05-06
**목적**: 사용자 화면에 표시되는 칼로리 숫자의 정확도를 23.8%(±20%) → 50%+로 끌어올리기 위한 코드 변경 계획 + 외부 자문(Gemini)을 위한 컨텍스트 정리.

---

## 1. NutriLens란

음식 사진 한 장을 GPT-4o Vision에 보내서 자동으로 음식을 인식하고 칼로리·단백질·탄수화물·지방을 출력해주는 한국 시장용 식단 분석 앱. 핵심 가치는 \"번거로운 수동 입력 없이 사진 한 장으로 끝\".

기술 스택:
- 백엔드: Python (Railway 호스팅), GPT-4o Vision API 호출
- 프론트: 단일 HTML/JS 파일 (서버에서 임베드)
- 음식 DB: `gold_foods.json` (1만건) + `CORE_FOODS` (Python 코드 안 hardcoded 355건, 식약처 기준 수동 검증)

분석 파이프라인:
1. 사용자 사진 업로드 → 서버
2. 서버가 base64로 인코딩 후 GPT-4o Vision 호출
3. GPT가 JSON으로 음식명·예상 양(g)·영양소 반환
4. 서버의 `match_with_db()` 함수가 GPT 응답을 한국 DB와 매칭하여 영양소 보정
5. 결과 사용자에게 표시

## 3. 핵심 진단

### 3.1 현재 GPT 프롬프트 (양 추정 가이드 부족)

`tools/food_analyzer.py`의 `SYSTEM_PROMPT` 중 양 추정 가이드는 단 한 줄:

```
2. 각 음식의 **예상 양(g)**을 추정하세요 (접시/그릇 크기 참고)
```

식기 크기·표준 1인분에 대한 정량적 정보가 없음. GPT는 \"감\"으로 추정해야 함.

### 3.2 매칭 로직 — GPT 양이 100% 우선됨

`match_with_db()`의 핵심 로직 발췌:

```python
gold_serving = gold_data.get('serving') or 100  # CORE_FOODS의 표준 1인분
ai_serving = food.get('estimated_serving_g')   # GPT 추정 양

if ai_serving:
    effective_serving = ai_serving           # ← GPT 양 무조건 사용
else:
    effective_serving = _estimate_realistic_serving(ai_name, gold_serving)

ratio = effective_serving / 100
food['calories_kcal'] = round(gold_data['cal'] * ratio, 1)
```

CORE_FOODS는 식약처 기준 표준 1인분(예: 김치 40g, 밥 210g)이 수동 검증되어 있는데, GPT가 \"양\"을 잘못 추정하면 그게 그대로 적용되어 CORE의 정확한 100g당 칼로리가 망가짐.

예: 김치 정답 = 17kcal/100g × 40g 표준 1인분 = 7kcal. 그런데 GPT가 \"100g\"으로 추정하면 17kcal로 표시 → 240% 과대.

## 4. 제안 변경

### 변경 A: SYSTEM_PROMPT에 \"식기·음식 표준 크기 참조표\" 추가 (~30줄)

GPT가 사진의 식기 크기와 음식 비율을 보고 양을 추정할 때 참조할 수 있는 구체적 숫자 제공:

```
## 표준 식기·음식 크기 참조표

### 그릇·접시 (한국식)
- 일반 밥공기: 직경 11cm, 깊이 5cm → 밥 1공기 약 210g
- 작은 반찬 종지: 직경 8cm → 반찬 30~50g
- 큰 사이드 그릇: 직경 13cm → 반찬 80~120g
- 큰 국·찌개 그릇: 직경 15cm, 깊이 7cm → 1인분 400~500g (국물 포함)

### 그릇·접시 (양식)
- 작은 디저트 접시 (15cm): 디저트 1조각, 100~150g
- 중간 접시 (22cm): 메인 요리 1인분, 200~350g
- 큰 접시 (28cm): 큰 메인, 400~500g
- 햄버거·샌드위치: 1개 200g 기준
- 피자 슬라이스: 1조각 약 100g, 큰 조각 150g

### 과일
- 사과·배: 1개 200g
- 바나나: 1개 120g
- 귤·키위: 1개 100g
- 딸기·블루베리: 한 줌 50~100g

### 음료
- 종이컵 (200ml): 음료 200g
- 머그컵 (300ml): 350g
- 캔 (355ml): 355g

### 분식·간식
- 떡·도넛 1개: 50~100g
- 쿠키 1개: 15~30g
- 초콜릿 1조각: 30~50g
- 감자튀김 1인분 (작은 컵): 100~150g

사진의 식기 크기·음식 비율을 보고 위 표를 참고해서 양을 추정하세요.
```

### 변경 B: match_with_db 양 보정 안전장치 (~10줄)

CORE_FOODS와 매칭됐는데 GPT 양이 표준의 **2배 초과 또는 0.5배 미만**이면 명백히 빗나간 추정으로 보고 CORE 표준 사용:

```python
if ai_serving:
    if gold_data.get('source') == 'core' and gold_serving > 0:
        # CORE_FOODS는 수동 검증된 1인분 — GPT가 너무 벗어나면 보정
        if ai_serving > gold_serving * 2.0 or ai_serving < gold_serving * 0.5:
            print(f"[양 보정] {ai_name}: GPT {ai_serving}g → CORE 표준 {gold_serving}g")
            effective_serving = gold_serving
        else:
            effective_serving = ai_serving
    else:
        effective_serving = ai_serving
else:
    effective_serving = _estimate_realistic_serving(ai_name, gold_serving)
```

### 변경 C: 회귀 테스트 단위 테스트 (`pre_deploy_test.py`)

위 보정 로직이 의도대로 동작하는지 자동 검증:
- GPT 양 1000g + CORE 표준 200g → 보정 → 200g 사용
- GPT 양 250g + CORE 표준 200g → 정상 → 250g 그대로
- GPT 양 50g + CORE 표준 200g → 보정 → 200g 사용

## 6. 검토한 대안 (채택 안 함)

| 대안 | 검토 이유 / 미채택 사유 |
|---|---|
| 사용자가 양 직접 조절 슬라이더 강화 | 이미 결과 화면에 음식 수정 기능 있음. 자동 정확도 향상이 더 가치 있음. (Phase 2 후보) |
| 사진 EXIF로 카메라 거리 추정 | 효과 미지수, 구현 복잡도 큼 |
| GPT-4o → GPT-4o-mini로 비용 절감 | 정확도 저하. 정확도 개선이 핵심이라 모델 다운그레이드는 역방향 |
| 다중 음식 사진에 식기 비율 분석 별도 단계 추가 | GPT 호출 1번 더 = 비용·지연 2배. 단일 호출 안에서 프롬프트로 해결 가능 |
| 사용자 키·체중 입력 받아 \"이 사람 1인분\" 추정 | 사용자 부담 증가. 진단 보고서의 다른 P1 작업 (초개인화 추천)에 적합 |
